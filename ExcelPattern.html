TA 之敗與 Excel 思考模式

緣起
TA 的原始構思是以網站形式將用戶 Excel 工作表「再電子化」(Reincarnation of Excel© worksheets) 的一種新嘗試， 以 ASP.NET WebForm 為基礎，輔以 CSS 及 Javascript 等工具。

計劃的第一部分工作紀錄，第二部份簽約預約；原定第三部分工作分派，由於協調問題而暫停開發。

何謂Excel 思考模式？
從用戶角度出發，要記錄任何資料，最方便莫過於使用 Excel 工作表，設定標題 (heading)、格式 (formating) 和公式 (formula)，便可立即投入使用。 工作表僅足夠目前使用即可，大幹快上，無需理會其他細枝節，轉過頭來增刪欄位亦非常容易。

較之於網站開發，先從編寫SQL 建立資料表開始，建立專案，資料輸入表單 (WebForm)，編寫代碼處理資料輸入/輸出，調試 (debug)， UAT (User Acceptance Test)，部署 (deploy) 等是一個完整開發週期 (Development Cycle)。 轉過頭來增刪欄位又是另一開發週期，還未計算由於修改所帶來的人手資料更新。

從手戶角度看 Excel 資料表易學、易用、可貯存成不同格式，不同版本。同樣事情發生在網站上可能是災難性的。

開發人員的困境
慣用 Excel 工作表解决問題的用戶會傾向於快速設定好工作表，盡快投入使用的工作模式，因其後續修改成本很低。 此與程式開發大不相同：

  1. 網頁介面的易用性無法與 Excel 工作表相比；
  2. 開發時間較制作 Excel 工作表數倍至數十倍以上；
  3. 工作表可簡單複制/傳送使用，網站 (程式及資料) 無法輕易 "dub" 多套出來用；
  4. 網站需要部署，源碼 (source code) 修改後需要回歸測試 (Regression Test)，無法快速應變；
  5. Excel 工作表不存在 one-to-one，one-to-many 等資料庫概念，只能增加欄位解決；
  6. 一般用戶可參與 Exel 工作表制作，綱站開發使用專屬工具，透過增加人手提速幾乎不可能；
  7. 用戶所看到的只是網站的表層呈現 (presentation layer)，維護和管理工作用戶無法得知；
  8. 使用網站應用程式的好處只有在長期運行，資料量變得很大時用戶才能體會。

後記
經常諗點解開發速度永遠追不上用戶需求？是計劃趕不上變化？抑或根本就冇計劃？
